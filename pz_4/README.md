# Практична робота №4

## Тезисно що було зроблено

-   Дослідження максимального обсягу пам'яті для `malloc(3)` на 32-х та
    64-х бітних системах.
-   Аналіз поведінки `malloc` при передачі від'ємних значень (**Integer
    Overflow**).
-   Дослідження граничних випадків: `malloc(0)`, `realloc` з `NULL` та
    `0`.
-   Аналіз викликів через `ltrace`.
-   Розробка власного алокатора пам'яті (аналог `malloc/free`) з
    використанням системних викликів `brk()` та `mmap()`.
-   Реалізація алгоритмів розбиття та об'єднання вільних блоків, аналіз
    впливу фрагментації пам'яті.
-   Адаптація коду під мінімальні середовища (**QEMU ARM64**):
    реалізовано пряме зв'язування з функціями ядра (`extern mmap`) в
    обхід відсутніх заголовних файлів (`sys/mmap.h`).

------------------------------------------------------------------------

# Програмні вимоги

-   **Середовище запуску:** Linux terminal (Ubuntu ARM64 / QEMU)\
-   **Компілятор:** GCC\
-   **Утиліта:** `ltrace`

------------------------------------------------------------------------

# Збірка та запуск

## 1. Скільки пам'яті може виділити malloc (Завдання 1)

Програма визначає розмір `size_t` та теоретичний максимум для алокації.\
На 64-бітній системі це **16 Ексабайт**.

### Компіляція та запуск (64-біт)

``` bash
gcc 1.c -o 1_64
./1_64
```

### Приклад результату

    Bits in size_t: 64
    Max size_t (bytes): 18446744073709551615
    Theoretical max allocation: 16 EB (Exabytes)

------------------------------------------------------------------------

## 2. Від'ємний аргумент та переповнення (Завдання 2)

При передачі від'ємного числа `int` у `malloc`, воно конвертується у
величезне беззнакове число `size_t`, що призводить до неможливості
виділення пам'яті.

### Запуск

``` bash
gcc 2.c -o 2
./2
```

### Результат

    xa: 2147483647, xb: 2
    num (signed int): -2
    num_sizet (unsigned): 18446744073709551614
    Didn't allocate 18446744073709551614 bytes

------------------------------------------------------------------------

## 3. malloc(0) (Завдання 3)

`malloc(0)` повертає унікальний вказівник, який можна безпечно передати
у `free()`.

### Запуск із ltrace

``` bash
gcc 3.c -o 3
ltrace ./3
```

### Результат

    malloc(0) = 0xaaaaab123450
    printf("Returned pointer: %p
    ", 0xaaaaab123450
    Returned pointer: 0xaaaaab123450
    ) = 33
    free(0xaaaaab123450) = <void>
    +++ exited (status 0) +++

------------------------------------------------------------------------

## 4. Помилка Use After Free (Завдання 4)

Демонстрація логічної помилки, коли перевірка `if (!ptr)` спрацьовує
неправильно, бо вказівник не було занулено після `free`.

У проблемній програмі пам'ять виділяється лише один раз, після чого
відбувається використання звільненої пам'яті.

У виправленій версії після `ptr = NULL` пам'ять коректно виділяється
щоразу.

### Запуск

``` bash
gcc 4.c -o 4
./4
```

### Результат

    --- Проблемний код ---
    Allocated: 0xaaaaab123450
    Помилка: ptr не NULL, але вказує на звільнену пам'ять: 0xaaaaab123450

    --- Виправлений код ---
    Allocated: 0xaaaaab123450
    Allocated: 0xaaaaab123450

------------------------------------------------------------------------

## 5. Невдалий realloc (Завдання 5)

Якщо `realloc` не може виділити пам'ять, він повертає `NULL`, але старий
вказівник залишається валідним.

### Запуск

``` bash
gcc 5.c -o 5
./5
```

### Результат

    Allocated 10 bytes at ptr: 0xaaaaab123450
    Didn't realloc to ptr1 (returned NULL)
    Original ptr is still valid: 0xaaaaab123450

------------------------------------------------------------------------

## 6. realloc з NULL або 0 (Завдання 6)

-   `realloc(NULL, size)` працює як `malloc(size)`
-   `realloc(ptr, 0)` працює як `free(ptr)`

### Запуск

``` bash
gcc 6.c -o 6
./6
```

### Результат

    Reallocated ptr with NULL: 0xaaaaab123450
    realloc with 0 size returned NULL (memory freed)

------------------------------------------------------------------------

## 7. Використання reallocarray (Завдання 7)

`reallocarray` безпечніша, бо перевіряє переповнення при множенні.

### Запуск

``` bash
gcc 7.c -o 7
./7
```

### Результат

    Callocated ptr: 0xaaaaab123450
    Reallocarray'd newptr: 0xaaaaab123450

------------------------------------------------------------------------

## 8. Власний алокатор пам'яті

Для малих блоків використовується `sbrk()`, для великих (\>128 KB) ---
`mmap()`.

Реалізовано: - розбиття великих блоків - об'єднання суміжних блоків

### Запуск

``` bash
gcc 8_allocator.c -o 8_allocator
./8_allocator
```

### Результат

    [brk] Allocated 1024 bytes at 0xaaaaab123000
    [brk] Allocated 512 bytes at 0xaaaaab123420
    [mmap] Allocated 200000 bytes at 0xffff98765000
    [free] Freed 1024 bytes at 0xaaaaab123000
    [brk] Splitting 1024-byte block at 0xaaaaab123000
    [free] Freed 512 bytes at 0xaaaaab123420
    [free] Freed 256 bytes at 0xaaaaab123000
    [merge] Free block is now 1536 bytes
    [munmap] Freed 200000 bytes at 0xffff98765000

------------------------------------------------------------------------

# Висновки

Було досліджено механізми керування динамічною пам'яттю в мові **C**.
Перевірено поведінку `malloc`, `free`, `realloc` у граничних випадках.

Продемонстровано: - вразливість **Use-After-Free** - проблему **Integer
Overflow**

Показано, що `reallocarray` є безпечнішою альтернативою.

Було реалізовано власний алокатор пам'яті з використанням `brk()` та
`mmap()`, що дозволило продемонструвати механізми керування купою та
важливість об'єднання вільних блоків для зменшення фрагментації.
