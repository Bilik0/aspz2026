# Практична робота №11

## Тезисно що було зроблено

- Написання програми для перехоплення сигналу критичного збою SIGSEGV.
- Отримання та логування адреси помилки (fault address), ідентифікатора процесу (PID) та стеку викликів (backtrace).
- Реалізація механізму автоматичного перезапуску програми після падіння з використанням execv та образу /proc/self/exe.
- Написання власних async-signal-safe функцій для конвертації чисел та вказівників у рядки без використання стандартної бібліотеки вводу-виводу.
- Забезпечення безпечного запису логів у файл через системні виклики open() та write().

---

## Програмні вимоги

- Середовище запуску - термінал Linux (Ubuntu).
- Компілятор - GCC. Обов'язкова наявність прапорців -g (для збереження символів налагодження) та -rdynamic (для експорту імен функцій у динамічну таблицю символів, щоб вони відображалися у backtrace).

# Збірка та запуск

## 1. Система логування та автоматичного відновлення (Завдання варіанту 1)

Було розроблено програму, яка діє як самовідновлювальна система.

Якщо стається критична помилка (наприклад, розіменування нульового вказівника), спеціальний обробник перехоплює управління. Він витягує PID, адресу збою з siginfo_t та поточний стек викликів, записуючи їх у лог-файл. Для безпеки замість printf або fopen використано прямий запис у файловий дескриптор, щоб уникнути дедлоків під час обробки переривання. Після збереження інформації програма намагається автоматично перезапустити саму себе.

### Компіляція та запуск:

```bash
gcc -g -rdynamic 1.c -o 1
~/lab11$ ./1
Програма запущена. PID: 3618
Генеруємо SIGSEGV...
Програма запущена. PID: 3618
Програму успішно відновлено після збою!
```

### Перегляд лог-файлу:

```bash
cat crash_log.txt
```

# Висновки

У роботі було розібрано механізми перехоплення критичних сигналів POSIX (на прикладі SIGSEGV) та використання структури siginfo_t для отримання точної адреси пам'яті, що викликала збій.

З точки зору системного програмування, було показано, як можна взаємодіяти з ядром Linux для отримання трасування стеку (backtrace) у момент аварії. Це критично важливо для пост-мортем аналізу програмного забезпечення, коли потрібно з'ясувати причину падіння без використання зовнішніх дебагерів на кшталт GDB.

Було суворо дотримано стандартів щодо async-signal-safety. Оскільки під час обробки сигналу стан купи (heap) та внутрішніх м'ютексів стандартної бібліотеки (наприклад, у функції malloc чи printf) може бути пошкоджений, було реалізовано власні функції конвертації та використано виключно безпечні системні виклики (write, open, backtrace_symbols_fd).

Також було приділено увагу відмовостійкості процесу. Замість того, щоб просто завершити роботу після зняття дампу, програма використовує системний виклик execv у поєднанні з лінуксовим псевдофайлом /proc/self/exe для завантаження та запуску свого ж образу наново. Це дозволяє створювати автономні сервіси, які здатні пережити тимчасові збої пам'яті та продовжити своє виконання.
