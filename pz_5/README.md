# Практична робота №5

## Тезисно що було зроблено

-   Написання програми на C, що містить критичну вразливість роботи з
    пам'яттю (**Out-of-bounds access**)
-   Дослідження структури пам'яті процесу, зокрема організації локальних
    змінних у стеку (**Stack**)
-   Моделювання ситуації "тихого" витоку даних (читання за межами масиву
    без виклику `segfault`)
-   Аналіз наслідків помилки, що призводить до несанкціонованої зміни
    логіки обчислень (підвищення привілеїв)

------------------------------------------------------------------------

# Програмні вимоги

-   **Середовище запуску:** термінал Linux (Ubuntu)
-   **Компілятор:** GCC (з прапорцями `-O0` та `-fno-stack-protector`)

# Збірка та запуск

## Читання за межами масиву (Варіант 1)

Написано програму, у якій послідовно оголошено локальну змінну (статус
адміністратора) та масив із трьох елементів.

Оскільки локальні змінні розташовуються у стеку (**Stack**) послідовно
одна за одною, звернення за межі масиву на один елемент (`arr[3]`) не
викликає помилки доступу до пам'яті (**Segmentation fault**), адже ця
ділянка все ще належить нашому процесу.

Замість аварійного завершення програма "тихо" зчитує значення сусідньої
змінної. Оскільки в мові C не перевіряються межі масивів під час
виконання, це призводить до логічної вразливості: програма читає дані
іншої змінної (в нашому випадку `0`), інтерпретує їх по-своєму, і це
призводить до несанкціонованого надання прав адміністратора.

Для успішної демонстрації уразливості компіляція проводилася з
прапорцями:

-   `-O0` --- вимкнення оптимізації компілятора
-   `-fno-stack-protector` --- вимкнення захисних механізмів стеку

------------------------------------------------------------------------

## Компіляція та запуск

``` bash
gcc -O0 -fno-stack-protector variant1.c -o variant1
./variant1
```

    Елементи масиву: 10, 20, 30

    Значення прочитане за межами масиву: 0

# Висновки

У роботі було розібрано особливості організації локальної пам'яті
(**Stack**) у програмах на мові **C**.

Практично доведено, що не всі помилки роботи з пам'яттю завершуються
миттєвим падінням програми (`segfault`). Помилка виходу за межі масиву
(**Out-of-bounds**) може протікати абсолютно непомітно для операційної
системи, викликаючи непередбачувану поведінку (**Undefined Behavior**)
або створюючи критичні вразливості безпеки.

Було змодельовано сценарій, коли відсутність вбудованої перевірки меж
масивів дозволила прочитати значення сусідньої змінної та змінити хід
виконання програми (отримати права адміністратора).

Це підкреслює необхідність:

-   жорсткого ручного контролю індексів масивів
-   використання інструментів динамічного аналізу пам'яті

Наприклад:

-   **Valgrind**
-   **AddressSanitizer**

Такі інструменти дозволяють виявляти приховані дефекти ще на етапі
розробки, що є критично важливим для написання безпечного системного
програмного забезпечення.
