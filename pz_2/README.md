# Практична робота №2

## Сегментація пам'яті та архітектура процесу

------------------------------------------------------------------------

## Опис роботи

Метою роботи є вивчення внутрішньої структури виконуваних файлів формату
**ELF** та розподілу пам'яті процесу в ОС **Linux**.

У роботі досліджено: - сегменти **Text**, **Data** та **BSS**; - роботу
**стека викликів** за допомогою `GDB`; - механізми **динамічного
виділення пам'яті (Heap)**.

------------------------------------------------------------------------

## Тезисно що було виконано

-   **Дослідження `time_t`**
    -   визначено розмір типу даних на 64-бітній архітектурі;
    -   змодельовано ситуацію переповнення часу.
-   **Аналіз сегментів пам'яті**
    -   досліджено вплив глобальних та локальних змінних;
    -   порівняно ініціалізовані та неініціалізовані дані.
-   **Робота з GDB**
    -   виконано аналіз стека процесу у стані блокування через системний
        виклик `pause()`.
-   **Варіант №1**
    -   визначено максимально можливий розмір сегмента купи (**Heap**).

------------------------------------------------------------------------

## Хід виконання та результати

### 1. Дослідження `time_t` (Завдання 2.1)

Програма показала, що тип `time_t` у сучасних системах має розмір **8
байт (64 біти)**.

Це усуває проблему **2038 року**, оскільки переповнення відбудеться лише
через мільярди років.

#### Результат виконання

``` bash
Size of time_t: 8 bytes
Max 32-bit time: Tue Jan 19 03:14:07 2038
After overflow (if 32-bit): Fri Dec 13 20:45:52 1901
```

------------------------------------------------------------------------

### 2. Аналіз сегментів виконуваного файлу (Завдання 2.2)

Було проведено вимірювання розмірів сегментів для різних конфігурацій
програми.

  ---------------------------------------------------------------------------------
  Етап      Зміна у коді             Результат (`size`)          Пояснення
  --------- ------------------------ --------------------------- ------------------
  Базовий   `printf("Hello");`       Стандартні значення         Код розміщується у
                                                                 Text

  BSS       `int arr[1000];`         BSS ↑ \~4000                Неініціалізовані
            (global)                                             дані не займають
                                                                 місця у файлі

  Data      `int arr[1000] = {1};`   Data ↑ \~4000               Дані зберігаються
                                                                 у виконуваному
                                                                 файлі

  Stack     `int arr[1000];` (local) Без змін                    Локальні змінні
                                                                 створюються під
                                                                 час виконання
  ---------------------------------------------------------------------------------

------------------------------------------------------------------------

### 3. Визначення адрес пам'яті (Завдання 2.3)

Було підтверджено розташування сегментів у віртуальному адресному
просторі:

-   **Text segment** --- виконувані інструкції.
-   **Data segment** --- ініціалізовані глобальні та статичні змінні.
-   **BSS segment** --- неініціалізовані глобальні змінні (заповнюються
    нулями).
-   **Stack** --- розташований у високих адресах та росте вниз.

------------------------------------------------------------------------

### 4. Дослідження стека процесу за допомогою GDB (Завдання 2.4)

Було виконано підключення до процесу за допомогою:

``` bash
gdb attach <pid>
```

Команда:

``` bash
bt
```

показала стек викликів:

    #0  __libc_pause
    #1  bar_is_now_closed()
    #2  bar()
    #3  foo()
    #4  main()

------------------------------------------------------------------------

### 5. Обґрунтування щодо IP та стека (Завдання 2.5)

Використання вершини стека замість **лічильника команд (IP)** є
неможливим, оскільки:

-   стек призначений для динамічних даних;
-   IP повинен вказувати на інструкції у сегменті **Text**;
-   сегмент коду має права лише **читання та виконання**, що забезпечує
    безпеку.

------------------------------------------------------------------------

## Завдання за варіантом №1

### Максимальний розмір купи (Heap)

Було реалізовано програму, яка виділяє пам'ять до моменту, поки:

``` c
malloc() == NULL
```

### Аналіз

Купа: - знаходиться у сегменті даних; - керується системними викликами
`brk()` або `mmap()`.

Максимальний розмір обмежується: - обсягом RAM; - розміром Swap; -
системними лімітами (`ulimit`).

------------------------------------------------------------------------

## 📊 Висновки

У ході роботи було встановлено:

-   сегмент **BSS** економить дисковий простір, зберігаючи лише розмір
    даних;
-   локальні змінні не впливають на розмір виконуваного файлу;
-   утиліти **size** та **gdb** є ефективними інструментами аналізу
    пам'яті процесу.
