# Практична робота №3

## Тезисно що було зроблено

-   Налаштування середовища **Docker** та дослідження поведінки
    контейнерів із різними рівнями привілеїв.
-   Дослідження системних лімітів (**ulimit**) на кількість відкритих
    файлів, розмір файлів, час процесора та розмір стека.
-   Збір системної статистики за допомогою інструменту профілювання
    **perf**.
-   Написання програм на **C**, що демонструють досягнення цих лімітів
    та обробку відповідних системних сигналів (**SIGXFSZ**,
    **SIGXCPU**).
-   Виконання стрес-тесту для перевірки максимальної кількості створених
    потоків (threads).

## Програмні вимоги

-   Середовище запуску: Ubuntu, контейнер у Docker
-   Компілятор: GCC
-   Утиліти: bash (ulimit), perf

## Збірка та запуск

### 1. Ліміти ulimit (Завдання 3.1)

Було проведено експеримент зі зміною ліміту на кількість відкритих
файлів у звичайному контейнері та контейнері з налаштованими правами.

**Обмежений режим:** Спроба підняти ліміт вище жорсткого значення (Hard
Limit) призвела до помилки, оскільки Docker захищає хостову систему:

``` text
root@9da2818decb8:/# ulimit -n 3000
root@9da2818decb8:/# ulimit -n 3001
bash: ulimit: open files: cannot modify limit: Operation not permitted
root@9da2818decb8:/# ulimit -n 2000
root@9da2818decb8:/# ulimit -n 3000
bash: ulimit: open files: cannot modify limit: Operation not permitted
```

**Режим з розширеними привілеями:** З відповідними дозволами зміна
ліміту пройшла успішно:

``` text
root@284a198db5ab:/# ulimit -n 2000
root@284a198db5ab:/# ulimit -n 3000
root@284a198db5ab:/# ulimit -n
3000
```

### 2. Аналіз продуктивності через perf (Завдання 3.2)

За допомогою утиліти perf було зібрано статистику виконання команди при
досягненні ліміту. Зафіксовано кількість помилок сторінок пам'яті
(page-faults) та витрачений процесорний час:

### 3. Обмеження розміру файлу (Завдання 3.3)

Програма записує результати кидків кубика у файл. Встановлено обробник
сигналу SIGXFSZ, що коректно перехоплює помилку та безпечно завершує
роботу при перевищенні ліміту розміру файлу:

``` text
root@284a198db5ab:/# ./task3_3 
Записано: 2
Записано: 5
Записано: 4
Записано: 2
Записано: 6
Записано: 2
Записано: 5

[Помилка] Перевищено ліміт розміру файлу (SIGXFSZ)! Завершення роботи.
```

### 4. Обмеження часу CPU (Завдання 3.4)

Програма генерує випадкові числа для лотереї (7 із 49 та 6 із 36).
Безперервний цикл створює навантаження на процесор, доки операційна
система не перериває виконання процесу при вичерпанні ліміту часу:

``` text
7 з 49: 20 43 28 23 37 31 15 
6 з 36: 9 18 33 27 30 5 
...
7 з 49: 35 3 20 11 31 30 21 
6 з 36: 26 34 4 15 28 9 
7 з 49: 9 43 32 41 24 29 48 
```

### 5. Копіювання файлів з обмеженням (Завдання 3.5)

Програма копіює вміст одного файлу в інший. При спробі скопіювати
великий файл, розмір якого перевищує встановлений внутрішній ліміт,
спрацьовує перехоплення сигналу і виводиться відповідне повідомлення:

``` text
root@284a198db5ab:/# ./task3_5 small.txt copy_small.txt
Файл успішно скопійовано.
root@284a198db5ab:/# ./task3_5 big.txt copy_big.txt
Файл успішно скопійовано.

[Помилка] Перевищено ліміт на розмір вихідного файлу!
```

### 6. Переповнення стека (Завдання 3.6)

Рекурсивна програма викликає сама себе, швидко заповнюючи виділену
пам'ять стека. При досягненні ліміту ulimit -s відбувається критична
помилка (Segmentation fault), оскільки ядро примусово вбиває процес:

``` text
Глибина рекурсії: 968
Глибина рекурсії: 969
Глибина рекурсії: 970
Глибина рекурсії: 971
Segmentation fault (core dumped)
```

### 7. Тест максимальної кількості потоків (Варіант 1)

Програма створює дочірні потоки за допомогою бібліотеки pthread у
нескінченному циклі. Оскільки потоки в Linux підпадають під ліміт
кількості процесів (ulimit -u), при його досягненні створення нових
потоків блокується:

``` text
Створено 3800 потоків...
Створено 3850 потоків...
Створено 3900 потоків...

[Помилка] Не вдалося створити потік! Досягнуто ліміт.
Всього створено потоків: 3944
```

## Висновки

У роботі було практично досліджено механізми управління ресурсами в ОС
Linux. За допомогою контейнеризації Docker було перевірено реакцію
системи на нестачу ресурсів: дискового простору, процесорного часу,
оперативної пам'яті (стека) та лімітів на створення потоків.

Було проаналізовано роль ядра ОС, яке за допомогою лімітів (ulimit)
забезпечує стабільність системи, запобігаючи монополізації ресурсів
окремими програмами. Також було продемонстровано роботу POSIX-сигналів
(зокрема, SIGXFSZ та SIGXCPU), за допомогою яких ядро повідомляє процеси
про порушення встановлених обмежень.

Практична реалізація завдань на мові C підтвердила важливість коректної
обробки системних викликів та сигналів. Це дозволяє створювати надійне
програмне забезпечення, яке не завершується аварійно при
непередбачуваній нестачі системних ресурсів, а здатне безпечно зупинити
роботу та поінформувати користувача про проблему.
